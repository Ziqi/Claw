## 二、四点补充意见

### 1. Nikto — 建议保留在旁路工具箱

导师判断 Nikto 是"上世纪产物"技术上成立。但在**CTF 靶场环境**中（DVWA、WebGoat 等古老靶机），Nikto 仍是最快的拿分工具。

**我的建议：**
- ❌ 不进主战流水线（同意导师）
- ✅ 放入 `make toolbox`，与导师推荐的 Nuclei 并存
- Nuclei 负责现代 Web 漏洞（YAML 模板、低噪声）
- Nikto 负责 CTF 古典靶机（现成可用、零配置）

### 2. Hydra — 建议限定 IoT 场景保留

导师指出域环境账号锁定风险**完全正确**。但 CatTeam 的侦察中已发现大量 IoT 设备（AirTunes、路由器），这些设备**通常没有锁定策略**。

**我的建议：**
- ❌ 严禁在域环境使用（同意导师的铁律）
- ✅ 作为 IoT 专用弱口令检测，限定 `PROFILE=iot`
- 脚本内加醒目红字警告 + scope.txt 白名单双重约束

### 3. MSF — auxiliary 模块值得保留

导师将 MSF 等同于"盲打 Exploit"只对了一半。MSF 的 **auxiliary 模块**（信息收集类）完全无害：

| 模块 | 用途 | 风险 |
|---|---|---|
| `auxiliary/scanner/smb/smb_version` | 精确识别 SMB 版本 | 🟢 零风险 |
| `auxiliary/scanner/vnc/vnc_none_auth` | 检测无密码 VNC | 🟢 零风险 |
| `auxiliary/scanner/http/title` | 批量抓网页标题 | 🟢 零风险 |

**我的建议：**
- ❌ exploit 模块绝不自动化（完全同意）
- ✅ 精选 3-5 个 auxiliary 模块作为 03-audit 的补充情报源
- 优先级确实低于 D/E/A/B/F（同意）

### 4. Auto 模式 — 建议分阶段实现

导师提出的 Auto 模式（自动判断 scope IP 是否同 L2 子网）思路优雅，但实现有坑：

- macOS `ifconfig` 输出格式随版本变化
- 多网卡场景（VPN + WiFi + 有线）判断复杂
- 边界情况多，容易出 Bug

**我的建议：**
- v4.0 先做 `passive | active` 两种**显式模式**
- v4.1 稳定后再加 `auto`（简单胜于聪明）

## 三、Nuclei 引入建议

Nuclei 战车内未预装，建议通过 Dockerfile v3 焊入：

```dockerfile
FROM my-kali-arsenal:v2
RUN apt-get update && apt-get install -y nuclei \
    && nuclei -update-templates \
    && apt-get clean && rm -rf /var/lib/apt/lists/*
```

**Nuclei 的定位：** 主战流水线的 Web 漏洞扫描器（替代 Nikto 在主链中的位置）
